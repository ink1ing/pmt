# 提示词评价模块设计

定期评价用户写给 AI 的提示词，产出一份浓缩、可直接执行的改进总结，并通过 Telegram 主动推送或静默写入用户指定路径。

实现位于 `Sources/PMT/AdviceEngine.swift`，复用既有的历史记录、调度、Telegram 与文件输出管道，分析时调用用户当前配置的同一个模型渠道（`ConfigStore.modelClient()`）。

## 定位与数据源

- 评价对象：用户自己撰写的原始提示词与语音输入，而非改写后的结果。
- 数据来源：本地历史 `~/Library/Application Support/PMT/advice-history.jsonl`，仅记录 `source = rewrite | dictation` 的原始输入（每条 ≥2 字、截断到 4000 字，总量上限 240 条）。
- 隐私：历史仅存本地，默认关闭；启用后仅把最近若干条片段发送给用户自配的模型；Telegram 只发最终总结，不发原始历史。

## 触发与调度

- 频率：手动 / 每天 / 每周（每周/每天在设定的时:分到点后触发一次）。
- 也可在设置面板手动"立即生成"。
- 单次执行加锁，避免重复并发生成。

## 分析 Agent 与专用提示词

- 使用用户已配置的模型渠道（OpenAI 兼容端点或 GitHub Copilot），不额外接入新模型。
- 仅替换 system prompt 为评价专用提示词；输出语言跟随 App 内中英开关。
- 输入载荷：最近约 80 条历史，格式 `[时间][来源] 文本`，整体截断到 12000 字。

## 输出契约

核心原则：提示词不是越详细越好，该详细则详细、该简略则简略；只在存在真实且重复出现的问题时才给建议。

结构：

1. 第一行：一句话总评，指出整体水平与当前最该改进的一个方向；若已足够好则写明"本期无需调整"。
2. 之后用 `1. 2. 3.` 列改进项；每条先点明观察到的重复模式，再给出下次可立即照做的具体改法，必要时附一个极简改写线索。

自适应详略与抗冗余（写入提示词的硬约束）：

- 已足够清晰有效则直接说明无需调整，绝不为凑数编造建议。
- 各条相互独立、主题不重叠，重叠则合并为一条。
- 禁止空泛建议（如只说"要更具体"却不给出具体做法）。
- 不复述原文、不泄露隐私、不解释分析过程、不寒暄。
- 按影响力从高到低排序，超出预算的低价值项直接舍弃。

详细程度 → 建议条数预算（`AdviceDetail.targetDescription`）：

| 档位 | 预算 |
| --- | --- |
| 极简 minimal | 最多 1 条；无明显问题只留一句总评 |
| 简短 brief | 最多 2 条，按影响力排序 |
| 标准 standard | 最多 3 条，可附 1 条做得好的点 |

## 两个出口

- 主动推送：Telegram（Bot Token + Chat ID，存于 Keychain），发送带日期头的总结。
- 静默存档：写入用户指定路径（默认 `~/Documents/PMT/advice.md`）。

## 产出文档形态

- 模型原始输出经轻量归一化：去除 Markdown 标题/强调/代码块/引用，保留编号与段落，单次报告封顶 1200 字。
- 文件为可累积的"总结文档"：每份报告带 `PMT 提示词评价 · 时间` 头部，新报告置顶、旧报告以分隔线保留，整体封顶 20000 字。
